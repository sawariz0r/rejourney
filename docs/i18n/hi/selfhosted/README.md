# स्व-होस्टिंग Rejourney

यह मार्गदर्शिका आधिकारिक **Docker Compose** स्टैक का उपयोग करके अपने स्वयं के सर्वर (आमतौर पर एक VPS या समर्पित मशीन) पर Rejourney चलाने वाले **कोई भी** के लिए है। आपको Rejourney के आंतरिक बुनियादी ढांचे या Kubernetes तक पहुंच की आवश्यकता नहीं है।

सेटअप के बाद आपको मिलेगा:

- आपके डोमेन पर एक **वेब डैशबोर्ड** (Let’s Encrypt के माध्यम से HTTPS)
- एक उपडोमेन पर एक **API** (डैशबोर्ड और मोबाइल SDK के लिए)
- किसी अन्य उपडोमेन पर एक **रिले को निगलना (अपलोड करना)।** (सत्र अपलोड आपके सर्वर से होते हैं, सीधे फोन से ऑब्जेक्ट स्टोरेज पर नहीं)
- **PostgreSQL**, **Redis**, और या तो **अंतर्निहित MinIO** या **आपका अपना S3-compatible भंडारण**
- पृष्ठभूमि **कार्यकर्ता** जो सत्र, अवधारण और अलर्ट की प्रक्रिया करती है (Rejourney के क्लाउड परिनियोजन में समान भूमिकाएँ)

नीचे दिए गए सभी आदेश मानते हैं कि क्लोनिंग के बाद आप **भंडार जड़** में हैं (फ़ोल्डर जिसमें `docker-compose.selfhosted.yml` है)।

## आपको पहले से क्या चाहिए

### सर्वर

- **ओएस:** Ubuntu 22.04+, Debian 12+, या कोई अन्य Linux जो Docker अच्छी तरह से चलता है
- **Docker:** 24 ​​या नया, **Docker Compose प्लगइन** के साथ (`docker compose version` को काम करना चाहिए)
- **संसाधन (अनुशंसित):** 4 vCPU, 8 जीबी रैम, 40 जीबी डिस्क (यदि आप कई रिकॉर्डिंग रखते हैं तो अधिक)
- **नेटवर्क:** पोर्ट **80** और **443** इंटरनेट के लिए खुले हैं (Let’s Encrypt HTTP चुनौती और HTTPS के लिए आवश्यक)

### डोमेन और DNS

आपको अपने नियंत्रण वाले **एक आधार डोमेन** की आवश्यकता है (उदाहरण के लिए `example.com`)। इंस्टॉलर चलाने से पहले, अपने सर्वर के सार्वजनिक आईपी पर इन होस्टनामों के **सभी** को इंगित करने वाले DNS **ए** (या **एएएए**) रिकॉर्ड बनाएं:

| होस्टनाम | उद्देश्य |
|----------|---------|
| `example.com` | डैशबोर्ड |
| `www.example.com` | डैशबोर्ड पर रीडायरेक्ट |
| `api.example.com` | API (और WebSocket जहां उपयोग किया जाता है) |
| `ingest.example.com` | रिले अपलोड करें (SDK API कॉन्फ़िगर होने के बाद स्वचालित रूप से इसका उपयोग करता है) |

`example.com` को अपने वास्तविक डोमेन से बदलें। प्रसार में कुछ मिनटों से लेकर घंटों तक का समय लग सकता है; TLS प्रमाणपत्र तब तक जारी नहीं होंगे जब तक DNS सही ढंग से हल न हो जाए।

### Let’s Encrypt

इंस्टॉल के दौरान आपसे **मेल पता** मांगा जाएगा। इसका उपयोग Let’s Encrypt से प्रमाणपत्र समाप्ति नोटिस के लिए किया जाता है।

### आपकी मशीन पर उपकरण

- रिपॉजिटरी को क्लोन करने के लिए `git`
- `openssl` (रहस्य उत्पन्न करने के लिए इंस्टॉल स्क्रिप्ट द्वारा उपयोग किया जाता है)
- एक शेल (bash ठीक है)

## पहली बार स्थापना

### 1. रिपॉजिटरी को क्लोन करें

```bash
git clone https://github.com/rejourneyco/rejourney.git
cd rejourney
```

डिफ़ॉल्ट शाखा पर बने रहें (या एक रिलीज़ टैग यदि परियोजना स्वयं-होस्टिंग के लिए दस्तावेज़ बनाती है)।

### 2. इंस्टॉलर चलाएँ

```bash
./scripts/selfhosted/deploy.sh install
```

स्क्रिप्ट होगी:

1. अपने **आधार डोमेन** के लिए पूछें (उदाहरण के लिए `example.com` - `https://` नहीं, कोई रास्ता नहीं)।
2. अपने **Let’s Encrypt ईमेल** के लिए पूछें।
3. **भंडारण** के लिए पूछें: अंतर्निहित **MinIO** (अनुशंसित) या **बाहरी S3-compatible** स्टोरेज (आप एंडपॉइंट, बकेट, क्षेत्र और कुंजियाँ दर्ज करेंगे)।
4. जेनरेट किए गए पासवर्ड और रहस्यों के साथ रेपो रूट में **`.env.selfhosted`** बनाएं। **अनुमतियाँ प्रतिबंधित करें** लागू होते हैं (`chmod 600`)।
5. **खींचो** ने कंटेनर छवियां प्रकाशित कीं (API, वेब, कर्मचारी, डेटाबेस, Traefik, आदि)।
6. **निर्माण** **बूटस्ट्रैप/माइग्रेशन** छवि **आपके क्लोन से** (इसमें डेटाबेस सेटअप स्क्रिप्ट शामिल हैं; इसे कंटेनर रजिस्ट्री से डाउनलोड नहीं किया गया है)।
7. डेटाबेस प्रारंभ करें, Redis, Traefik, और (यदि चुना गया हो) MinIO।
8. बूटस्ट्रैप चलने से पहले कॉन्फ़िगर किए गए `DATABASE_URL` का उपयोग करके डेटाबेस कनेक्टिविटी को सत्यापित करें।
9. एक-शॉट **बूटस्ट्रैप** कंटेनर चलाएँ: डेटाबेस स्कीमा, वैकल्पिक पहली बार बीज, और डेटाबेस में भंडारण कॉन्फ़िगरेशन।
10. API प्रारंभ करें, रिले, डैशबोर्ड और वर्कर्स अपलोड करें।

पहली स्थापना में कई मिनट लग सकते हैं (छवि खींच और बूटस्ट्रैप)।

### 3. `.env.selfhosted` को सुरक्षित रखें

यह फ़ाइल आपकी तैनाती के लिए **सभी रहस्य** रखती है (डेटाबेस, Redis, JWT, स्टोरेज एन्क्रिप्शन, MinIO क्रेडेंशियल यदि उपयोग किया जाता है, आदि)। **इसे वापस लें** को एक सुरक्षित स्थान पर (पासवर्ड मैनेजर, एन्क्रिप्टेड बैकअप)। यदि आप इसे खो देते हैं, तो आप संग्रहीत क्रेडेंशियल्स को डिक्रिप्ट करने या उसी परिनियोजन को फिर से बनाने की क्षमता खो सकते हैं।

इसे git के लिए प्रतिबद्ध न करें (इसे `.gitignore` द्वारा अनदेखा किया जाना चाहिए)।

## स्थापना के बाद

### यूआरएल

इंस्टॉलर यूआरएल प्रिंट करता है। सामान्य तौर पर:

- **डैशबोर्ड:** `https://<your-base-domain>`
- **API:** `https://api.<your-base-domain>`
- **निगलना:** `https://ingest.<your-base-domain>`

`www.<your-base-domain>` डैशबोर्ड पर रीडायरेक्ट करता है।

### स्टैक सत्यापित करें

```bash
./scripts/selfhosted/deploy.sh status
```

आपको कंटेनर चलते हुए दिखना चाहिए; `api` और `ingest-upload` को थोड़े समय के बाद **स्वस्थ** बन जाना चाहिए।

### सबसे पहले लॉग इन करें और रिकॉर्डिंग का परीक्षण करें

1. ब्राउज़र में डैशबोर्ड खोलें.
2. एक खाता और एक प्रोजेक्ट बनाएं.
3. अपने ऐप के Rejourney SDK को अपने **API यूआरएल** के साथ कॉन्फ़िगर करें (नीचे [SDK कॉन्फ़िगरेशन](#configuring-your-mobile-app) देखें)।
4. एक छोटा सत्र रिकॉर्ड करें और पुष्टि करें कि यह रीप्ले में दिखाई देता है।

यदि सत्र रीप्ले में कभी दिखाई नहीं देते हैं, तो [समस्या निवारण](/docs/selfhosted/troubleshooting) देखें (रिले अपलोड करें और वर्कर लॉग इन्जेस्ट करें)।

## दिन-प्रतिदिन के कार्य

ये सभी रेपो रूट से चलते हैं।

| कार्रवाई | आदेश |
|--------|---------|
| सेवा स्थिति | `./scripts/selfhosted/deploy.sh status` |
| सभी लॉग का पालन करें | `./scripts/selfhosted/deploy.sh logs` |
| एक सेवा के लिए लॉग | `./scripts/selfhosted/deploy.sh logs api` (`api` को `web`, `ingest-upload`, `ingest-worker`, आदि से बदलें) |
| **उन्नत करना** छवियाँ और बूटस्ट्रैप पुनः चलाएँ | `./scripts/selfhosted/deploy.sh update` |
| सब कुछ **बिना** डेटा हटाना बंद करें | `./scripts/selfhosted/deploy.sh stop` |
| **रीसेट करें** कंटेनर और वॉल्यूम (विनाशकारी) | `./scripts/selfhosted/deploy.sh reset` |

**`update`** नई छवियां खींचता है (जहां लागू हो), आपके वर्तमान क्लोन से बूटस्ट्रैप छवि को फिर से बनाता है, स्टैक को पुनरारंभ करता है, और बूटस्ट्रैप को फिर से चलाता है ताकि डेटाबेस स्कीमा और स्टोरेज सेटिंग्स आपके `.env.selfhosted` के साथ संरेखित रहें। यह **नहीं** Postgres या ऑब्जेक्ट स्टोरेज वॉल्यूम को मिटा देता है।

बूटस्ट्रैप से पहले, `install` और `update` दोनों कॉन्फ़िगर क्रेडेंशियल्स के साथ डेटाबेस कनेक्टिविटी को मान्य करते हैं। यदि क्रेडेंशियल निरंतर Postgres डेटा से मेल नहीं खाते हैं, तो बूटस्ट्रैप में बाद में विफल होने के बजाय पुनर्प्राप्ति मार्गदर्शन के साथ तैनाती जल्दी बंद हो जाती है।

**`stop`** केवल कंटेनरों को रोकता है; Docker **संस्करणों** (Postgres डेटा, MinIO डेटा, आदि) तब तक बने रहते हैं जब तक आप उन्हें स्पष्ट रूप से हटा नहीं देते।

**`reset`** एक पुष्टिकरण संकेत के बाद स्व-होस्टेड कंटेनर और Docker वॉल्यूम (`pgdata`, `redisdata`, `miniodata`, `traefik-certs`) को हटा देता है। यह `.env.selfhosted` गायब होने पर भी MinIO प्रोफ़ाइल कंटेनर को फाड़ देता है, इसलिए पुराना MinIO डेटा अगली स्थापना को अवरुद्ध नहीं करता है। इसका उपयोग केवल तभी करें जब आप पूरी तरह से ताज़ा इंस्टालेशन चाहते हों।

## भंडारण: MinIO बनाम बाहरी S3

### अंतर्निहित MinIO (डिफ़ॉल्ट)

- एकल सर्वर के लिए सबसे आसान: ऑब्जेक्ट स्टोरेज **Docker के अंदर** चलता है और डिफ़ॉल्ट रूप से सार्वजनिक इंटरनेट के संपर्क में नहीं आता है।
- सत्र बाइट्स **निगलना-अपलोड करें** सेवा द्वारा लिखे गए हैं; उपकरणों को सीधे MinIO तक पहुंचने की आवश्यकता नहीं है।
- इंस्टालेशन के दौरान बकेट निर्माण का कार्य संभाला जाता है।

### बाहरी S3-compatible भंडारण

AWS S3, Cloudflare R2, हेट्ज़नर ऑब्जेक्ट स्टोरेज, वसाबी, या किसी S3-compatible API का उपयोग करें। इंस्टॉल के दौरान आप एंडपॉइंट यूआरएल, बकेट, क्षेत्र और एक्सेस कुंजियाँ प्रदान करते हैं।

एंडपॉइंट यूआरएल शैलियों के उदाहरण (आपके प्रदाता के दस्तावेज़ आधिकारिक हैं):

- AWS: `https://s3.<region>.amazonaws.com`
- Cloudflare R2: `https://<account-id>.r2.cloudflarestorage.com`
- हेट्ज़नर: `https://<location>.your-objectstorage.com`

यदि आप डाउनलोड के लिए **अलग सार्वजनिक यूआरएल** जोड़ते हैं, तो `.env.selfhosted` में `S3_PUBLIC_ENDPOINT` सेट करें और `./scripts/selfhosted/deploy.sh update` चलाएं।

## महत्वपूर्ण कॉन्फ़िगरेशन (`.env.selfhosted`)

इंस्टॉलर इस फ़ाइल को जनरेट करता है। विशिष्ट चर में शामिल हैं:

- **डोमेन और सार्वजनिक यूआरएल:** `BASE_DOMAIN`, `DASHBOARD_DOMAIN`, `API_DOMAIN`, `INGEST_DOMAIN`, `PUBLIC_*_URL`
- **डेटाबेस:** `DATABASE_URL` (Compose के अंदर `postgres` सेवा पर इंगित करता है)
- **Redis:** `REDIS_URL`
- **भंडारण:** `STORAGE_BACKEND`, `S3_*`, और वैकल्पिक रूप से `MINIO_*`
- **सुरक्षा:** `JWT_SECRET`, `JWT_SIGNING_KEY`, `INGEST_HMAC_SECRET`, `STORAGE_ENCRYPTION_KEY`

वैकल्पिक एकीकरण (यदि अप्रयुक्त हो तो खाली छोड़ें): Stripe, SMTP, GitHub OAuth, आदि।

**भंडारण या डोमेन-संबंधित मान बदलना:** संपादित करें `.env.selfhosted`, फिर चलाएँ:

```bash
./scripts/selfhosted/deploy.sh update
```

## डेटाबेस सेटअप कैसे काम करता है (पहला बूट बनाम बाद का अपडेट)

आपको आमतौर पर **नहीं** को हाथ से SQL चलाने की आवश्यकता होती है। **बूटस्ट्रैप** कंटेनर इसे संभालता है।

- **बिल्कुल नया खाली डेटाबेस:** स्टैक कोड से वर्तमान स्कीमा को लागू करता है, फिर रिकॉर्ड करता है कि कौन से माइग्रेशन संस्करण पहले से ही संतुष्ट हैं इसलिए भविष्य के अपडेट केवल **नया** माइग्रेशन लागू करते हैं।
- **मौजूदा डेटाबेस (पहले से ही प्रारंभ):** केवल **लंबित** माइग्रेशन लागू होते हैं। प्रत्येक `update` पर आपका डेटा स्क्रैच से दोबारा नहीं बनाया जाता है।
- यदि डेटाबेस **पहले से ही टेबल हैं** है लेकिन माइग्रेशन इतिहास तालिका **गुम या खाली** है (उदाहरण के लिए आंशिक पुनर्स्थापना), तो आकस्मिक क्षति से बचने के लिए बूटस्ट्रैप **एक त्रुटि के साथ रुक जाता है** करें। उन्नत पुनर्प्राप्ति विकल्प [समस्या निवारण](/docs/selfhosted/troubleshooting) में प्रलेखित हैं।

## Apple सिलिकॉन और ARM सर्वर

**एआरएम64** मशीनों (कई मैक, कुछ क्लाउड इंस्टेंस) पर, तैनाती स्क्रिप्ट छवि खींचने के लिए `DOCKER_DEFAULT_PLATFORM=linux/amd64` सेट करती है जब आपने इसे स्वयं सेट नहीं किया है, इसलिए पूर्वनिर्मित छवियां जो केवल `amd64` प्रकाशित करती हैं, अभी भी चलती हैं। यदि आपको भिन्न व्यवहार की आवश्यकता है, तो स्क्रिप्ट चलाने से पहले अपने वातावरण में `DOCKER_DEFAULT_PLATFORM` सेट करें।

**बूटस्ट्रैप** छवि हमेशा क्लोन रिपॉजिटरी से **आपकी मशीन पर बनाया गया** होती है, इसलिए यह हमेशा आपके चेकआउट से मेल खाती है।

## Docker में क्या चलता है (अवलोकन)

- **Traefik:** HTTPS प्रमाणपत्र और डैशबोर्ड पर रूटिंग, API, और होस्टनाम ग्रहण करें।
- **Postgres / Redis:** एप्लिकेशन डेटा और कतारें।
- **MinIO:** वैकल्पिक आंतरिक ऑब्जेक्ट भंडारण।
- **API:** मुख्य HTTP API।
- **निगलना-अपलोड:** अपलोड रिले ट्रैफ़िक के लिए समर्पित सेवा।
- **वेब:** डैशबोर्ड स्थिर यूआई।
- **श्रमिक:** प्रक्रिया अंतर्ग्रहण कतारें, कलाकृतियों को फिर से चलाना, सत्र जीवनचक्र, निर्धारित अवधारण-शैली कार्य और अलर्ट।

इस स्टैक में **नहीं** अलग बिलिंग बैच कार्यकर्ता है; जब आप कुंजियाँ कॉन्फ़िगर करते हैं तो बिलिंग एकीकरण Stripe और API द्वारा संचालित होता है।

## आपके मोबाइल ऐप को कॉन्फ़िगर किया जा रहा है

SDK को **आपका** API होस्ट पर इंगित करें (`API_DOMAIN` / `PUBLIC_API_URL` से मेल खाना चाहिए)।

### React Native उदाहरण

```ts
import { initRejourney, startRejourney } from '@rejourneyco/react-native';

initRejourney('pk_live_your_public_key', {
  apiUrl: 'https://api.example.com',
});

startRejourney();
```

अपने वास्तविक API URL का उपयोग करें। जब सर्वर सही ढंग से कॉन्फ़िगर किया जाता है तो अपलोड URL स्वचालित रूप से `ingest.<your-domain>` के लिए प्राप्त होते हैं।
