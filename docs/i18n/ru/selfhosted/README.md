# Самостоятельный хостинг Rejourney

Это руководство предназначено для **любой**, запускающего Rejourney на собственном сервере (обычно на одном VPS или выделенном компьютере) с использованием официального стека **Docker Compose**. Вам не нужен доступ к внутренней инфраструктуре Rejourney или Kubernetes.

После настройки вы получаете:

- **веб-панель** в вашем домене (HTTPS через Let’s Encrypt)
- **API** на субдомене (для информационной панели и мобильного SDK)
- **реле приема (загрузки)** на другом субдомене (загрузка сеанса происходит через ваш сервер, а не напрямую с телефонов в хранилище объектов)
- **PostgreSQL**, **Redis** и **встроенный MinIO** или **ваше собственное хранилище S3-compatible**
- Фоновый **рабочие**, который обрабатывает сеансы, хранение и оповещения (те же роли, что и в облачном развертывании Rejourney)

Все приведенные ниже команды предполагают, что после клонирования вы находитесь в **корень репозитория** (папка, содержащая `docker-compose.selfhosted.yml`).

---

## Что вам нужно заранее

### Сервер

- **ОС:** Ubuntu 22.04+, Debian 12+ или другой Linux, который хорошо работает с Docker
- **Docker:** 24 ​​или новее, с **Плагин Docker Compose** (`docker compose version` должен работать)
- **Ресурсы (рекомендуется):** 4 виртуальных ЦП, 8 ГБ ОЗУ, диск 40 ГБ (больше, если вы храните много записей)
- **Сеть:** Порты **80** и **443**, открытые для Интернета (требуются для вызовов Let’s Encrypt HTTP и HTTPS)

### Домен и DNS

Вам нужен **один базовый домен**, которым вы управляете (например, `example.com`). Перед запуском установщика создайте записи DNS **А** (или **АААА**), указывающие **все** этих имен хостов на общедоступный IP-адрес вашего сервера:

| Имя хоста | Цель |
|----------|---------|
| `example.com` | Панель управления |
| `www.example.com` | Перенаправляет на панель управления |
| `api.example.com` | API (и WebSocket, где используется) |
| `ingest.example.com` | Реле загрузки (SDK использует это автоматически после настройки API) |

Замените `example.com` на свой реальный домен. Распространение может занять от нескольких минут до часов; Сертификаты TLS не будут выдаваться до тех пор, пока DNS не будет правильно решен.

### Let’s Encrypt

Во время установки вам будет предложено ввести **Адрес электронной почты**. Он используется для уведомлений об истечении срока действия сертификата от Let’s Encrypt.

### Инструменты на вашей машине

- `git` для клонирования репозитория
- `openssl` (используется сценарием установки для создания секретов)
- Оболочка (bash подойдет)

## Первая установка

### 1. Клонировать репозиторий

```bash
git clone https://github.com/rejourneyco/rejourney.git
cd rejourney
```

Оставайтесь в ветке по умолчанию (или в теге выпуска, если проект документирует ветку для самостоятельного размещения).

### 2. Запустите установщик

```bash
./scripts/selfhosted/deploy.sh install
```

Скрипт будет:

1. Попросите свой **базовый домен** (например, `example.com` — не `https://`, без пути).
2. Попросите свой **Let’s Encrypt электронная почта**.
3. Запросите **хранилище**: встроенное хранилище **MinIO** (рекомендуется) или **внешний S3-compatible** (вы должны ввести конечную точку, корзину, регион и ключи).
4. Создайте **`.env.selfhosted`** в корне репо с сгенерированными паролями и секретами. **Ограничить разрешения** применяются (`chmod 600`).
5. **Тянуть** опубликованные образы контейнеров (API, веб, рабочие процессы, базы данных, Traefik и т. д.).
6. **Строить** образ **начальная загрузка/миграция** **из твоего клона** (содержит скрипты настройки базы данных, не загружается из реестра контейнера).
7. Запустите базы данных Redis, Traefik и (если выбрано) MinIO.
8. Перед запуском начальной загрузки проверьте подключение к базе данных с помощью настроенного `DATABASE_URL`.
9. Запустите одноразовый контейнер **самозагрузка**: схема базы данных, необязательное первое начальное значение и конфигурация хранилища в базе данных.
10. Запустите API, загрузите реле, панель мониторинга и рабочие процессы.

Первая установка может занять несколько минут (извлечение образа и загрузка).

### 3. Защитите `.env.selfhosted`

Этот файл содержит **все секреты** для вашего развертывания (база данных, Redis, JWT, шифрование хранилища, учетные данные MinIO, если они используются, и т. д.). **Сделайте резервную копию** в безопасное место (менеджер паролей, зашифрованная резервная копия). Если вы потеряете его, вы можете потерять возможность расшифровать сохраненные учетные данные или восстановить то же развертывание.

Не фиксируйте его в git (`.gitignore` должен игнорировать его).

## После установки

### URL-адреса

Установщик распечатает URL-адреса. В общем:

- **Панель управления:** `https://<your-base-domain>`
- **API:** `https://api.<your-base-domain>`
- **Всасывание:** `https://ingest.<your-base-domain>`

`www.<your-base-domain>` перенаправляет на панель управления.

### Проверьте стек

```bash
./scripts/selfhosted/deploy.sh status
```

Вы должны увидеть работающие контейнеры; `api` и `ingest-upload` через короткое время должны стать **здоровый**.

### Первый вход в систему и тестовая запись

1. Откройте панель управления в браузере.
2. Создайте аккаунт и проект.
3. Настройте Rejourney SDK вашего приложения с помощью **API URL-адрес** (см. [Конфигурация SDK](#configuring-your-mobile-app) ниже).
4. Запишите короткий сеанс и убедитесь, что он появился в Replay.

Если сеансы никогда не отображаются в Replay, см. раздел [Устранение неполадок](/docs/selfhosted/troubleshooting) (загрузка журналов ретрансляции и приема рабочих процессов).

## Ежедневные операции

Все они запускаются из корня репо.

| Действие | Команда |
|--------|---------|
| Статус услуги | `./scripts/selfhosted/deploy.sh status` |
| Следить за всеми журналами | `./scripts/selfhosted/deploy.sh logs` |
| Логи для одного сервиса | `./scripts/selfhosted/deploy.sh logs api` (замените `api` на `web`, `ingest-upload`, `ingest-worker` и т. д.) |
| **Обновление** образы и повторная загрузка | `./scripts/selfhosted/deploy.sh update` |
| Остановить все **без** удаление данных | `./scripts/selfhosted/deploy.sh stop` |
| **Перезагрузить** контейнеры и тома (разрушительные) | `./scripts/selfhosted/deploy.sh reset` |

**`update`** извлекает новые образы (где это применимо), восстанавливает загрузочный образ из текущего клона, перезапускает стек и снова запускает загрузку, чтобы схема базы данных и настройки хранилища оставались согласованными с вашим `.env.selfhosted`. **нет** стирает Postgres или тома хранилища объектов.

Перед начальной загрузкой и `install`, и `update` проверяют подключение к базе данных с настроенными учетными данными. Если учетные данные не соответствуют сохраненным данным Postgres, развертывание прекращается раньше с инструкциями по восстановлению, а не завершается сбоем позже при начальной загрузке.

**`stop`** останавливает только контейнеры; Docker **объемы** (данные Postgres, данные MinIO и т. д.) остаются до тех пор, пока вы не удалите их явно.

**`reset`** удаляет локальные контейнеры и тома Docker (`pgdata`, `redisdata`, `miniodata`, `traefik-certs`) после запроса на подтверждение. Он также удаляет контейнеры профиля MinIO, даже если `.env.selfhosted` отсутствует, поэтому устаревшие данные MinIO не блокируют следующую установку. Используйте это только в том случае, если вам нужна полностью новая установка.

## Хранилище: MinIO против внешнего S3

### Встроенный MinIO (по умолчанию)

- Проще всего для одного сервера: объектное хранилище работает под управлением **внутри Docker** и по умолчанию не доступно общедоступному Интернету.
- Байты сеанса записываются службой **загрузка-загрузка**; устройствам не обязательно напрямую подключаться к MinIO.
- Создание сегмента выполняется во время установки.

### Внешнее хранилище S3-compatible

Используйте AWS S3, Cloudflare R2, Hetzner Object Storage, Wasabi или любой S3-compatible API. Во время установки вы указываете URL-адрес конечной точки, корзину, регион и ключи доступа.

Примеры стилей URL-адресов конечных точек (документы вашего провайдера являются авторитетными):

- AWS: `https://s3.<region>.amazonaws.com`
- Cloudflare R2: `https://<account-id>.r2.cloudflarestorage.com`
- Хетцнер: `https://<location>.your-objectstorage.com`

Если вы добавляете **отдельный общедоступный URL-адрес** для загрузки, установите `S3_PUBLIC_ENDPOINT` в `.env.selfhosted` и запустите `./scripts/selfhosted/deploy.sh update`.

## Важная конфигурация (`.env.selfhosted`)

Установщик создает этот файл. Типичные переменные включают в себя:

- **Домены и общедоступные URL-адреса:** `BASE_DOMAIN`, `DASHBOARD_DOMAIN`, `API_DOMAIN`, `INGEST_DOMAIN`, `PUBLIC_*_URL`
- **База данных:** `DATABASE_URL` (указывает на сервис `postgres` внутри Compose)
- **Redis:** `REDIS_URL`
- **Хранилище:** `STORAGE_BACKEND`, `S3_*` и дополнительно `MINIO_*`
- **Безопасность:** `JWT_SECRET`, `JWT_SIGNING_KEY`, `INGEST_HMAC_SECRET`, `STORAGE_ENCRYPTION_KEY`

Дополнительные интеграции (оставьте пустыми, если не используется): Stripe, SMTP, GitHub OAuth и т. д.

**Изменение значений, связанных с хранилищем или доменом:** отредактируйте `.env.selfhosted`, затем запустите:

```bash
./scripts/selfhosted/deploy.sh update
```

## Как работает настройка базы данных (первая загрузка и последующие обновления)

Обычно вам нужно запустить SQL вручную. Это обрабатывает контейнер **самозагрузка**.

- **Совершенно новая пустая база данных:** стек применяет текущую схему из кода, а затем записывает, какие версии миграции уже удовлетворены, поэтому будущие обновления применяют только миграции **новый**.
- **Существующая база данных (уже инициализирована):** Применяются только миграции **в ожидании**. Ваши данные не восстанавливаются с нуля на каждом `update`.
- Если база данных **уже есть таблицы**, но таблица истории миграции — **отсутствует или пуст** (например, частичное восстановление), загрузите **останавливается с ошибкой**, чтобы избежать случайного повреждения. Дополнительные параметры восстановления описаны в разделе [Устранение неполадок](/docs/selfhosted/troubleshooting).

## Серверы Apple Silicon и ARM

На компьютерах **ARM64** (многие компьютеры Mac, некоторые облачные экземпляры) сценарий развертывания устанавливает `DOCKER_DEFAULT_PLATFORM=linux/amd64` для извлечения образа, если вы не установили его самостоятельно, поэтому готовые образы, которые публикуют только `amd64`, все равно выполняются. Если вам нужно другое поведение, установите `DOCKER_DEFAULT_PLATFORM` в своей среде перед запуском сценария.

Образ **самозагрузка** всегда представляет собой **построен на вашей машине** из клонированного репозитория, поэтому он всегда соответствует вашему оформлению заказа.

## Что работает в Docker (обзор)

- **Traefik:** HTTPS сертификаты и маршрутизация на панель мониторинга API и прием имен хостов.
- **Postgres / Redis:** Данные приложений и очереди.
- **MinIO:** Дополнительное внутреннее объектное хранилище.
- **API:** Основной HTTP API.
- **загрузка-загрузка:** Выделенный сервис для загрузки ретрансляционного трафика.
- **сеть:** Статический пользовательский интерфейс информационной панели.
- **Рабочие:** Обработка очередей приема, артефактов воспроизведения, жизненного цикла сеанса, запланированной работы в стиле хранения и оповещений.

В этом стеке есть отдельный пакетный исполнитель выставления счетов **нет**; Интеграция выставления счетов осуществляется с помощью Stripe и API при настройке ключей.

## Настройка вашего мобильного приложения

Направьте SDK на хост **твой** API (должен соответствовать `API_DOMAIN` / `PUBLIC_API_URL`).

### Пример React Native

```ts
import { initRejourney, startRejourney } from '@rejourneyco/react-native';

initRejourney('pk_live_your_public_key', {
  apiUrl: 'https://api.example.com',
});

startRejourney();
```

Используйте свой реальный URL-адрес API. URL-адреса загрузки создаются для `ingest.<your-domain>` автоматически, если сервер настроен правильно.

## Резервные копии

Как минимум сделайте резервную копию **PostgreSQL**, **`.env.selfhosted`** и (если вы используете встроенный MinIO) **данные хранилища объектов**.

```bash
./scripts/selfhosted/backup.sh
./scripts/selfhosted/backup.sh --full
```

Подробности: [Резервное копирование и восстановление](/docs/selfhosted/backup-recovery).
